基本的网络搜索
让我们从我们以前去过的地方开始：在浏览器的地址栏中输入“www.github.com”并查看页面加载。

尽管这笔交易看起来很简单，但幕后却发生了大量的魔术。让我们直接进入它。
定义 Web 的各个部分
了解网络是令人生畏的，因为有很多行话。不幸的是，其中一些行话对于理解本文的其余部分至关重要。
如果您想了解万维网的秘密，以下是需要了解的最重要的术语：
客户：在计算机上运行并连接到 Internet 的应用程序，例如 Chrome 或 Firefox。它的主要作用是进行用户交互并将其转换为对另一台称为 Web 服务器的计算机的请求。虽然我们通常使用浏览器来访问 Web，但您可以将整个计算机视为客户端-服务器模型的“客户端”部分。每台客户端计算机都有一个唯一的地址，称为 IP 地址，其他计算机可以使用该地址来标识它。
服务器：连接到 Internet 且具有 IP 地址的计算机。服务器等待来自其他机器（例如客户端）的请求并响应它们。与同样具有 IP 地址的计算机（即客户端）不同，服务器安装并运行了特殊的服务器软件，该软件告诉它如何响应来自浏览器的传入请求。Web 服务器的主要功能是存储、处理网页并将其交付给客户端。服务器有多种类型，包括 Web 服务器、数据库服务器、文件服务器、应用程序服务器等。（在这篇文章中，我们谈论的是 Web 服务器。
IP地址：Internet 协议地址。TCP/IP 网络上设备（计算机、服务器、打印机、路由器等）的数字标识符。互联网上的每台计算机都有一个 IP 地址，用于识别其他计算机并与之通信。IP 地址有四组数字，用小数点分隔（例如 244.155.65.2）。这称为“逻辑地址”。为了在网络中定位设备，TCP/IP 协议软件将逻辑 IP 地址转换为物理地址。此物理地址（即 MAC 地址）内置于您的硬件中。
所属单位：互联网服务提供商。ISP是客户端和服务器之间的中间人。对于典型的房主来说，ISP通常是一家“有线电视公司”。当您的浏览器收到您访问www.github.com的请求时，它不知道在哪里查找www.github.com。因此，ISP 的工作是进行 DNS（域名系统）查找，以询问您尝试访问的站点配置为哪个 IP 地址。

DNS：域名系统。一个分布式数据库，用于跟踪计算机的域名及其在互联网上的相应IP地址。现在不用担心“分布式数据库”是如何工作的：只要知道DNS的存在，这样用户就可以输入www.github.com而不是IP地址。

域名：用于标识一个或多个IP 地址。用户使用域名（例如www.github.com）访问互联网上的网站。当您在浏览器中输入域名时，DNS 会使用它来查找该给定网站的相应 IP 地址。

TCP/IP协议：传输控制协议/互联网协议。使用最广泛的通信协议。“协议”只是做某事的一套标准规则。TCP/IP 被用作通过网络传输数据的标准。
端口号：一个 16 位整数，用于标识服务器上的特定端口，并始终与 IP 地址相关联。它是一种识别服务器上网络请求可以转发到的特定进程的方法。
主机：连接到网络的计算机 - 它可以是客户端、服务器或任何其他类型的设备。每个主机都有一个唯一的 IP 地址。对于像 www.google.com 这样的网站，主机可以是为网站页面提供服务的 Web 服务器。主机和服务器之间经常存在一些混淆，但请注意它们是两个不同的东西。服务器是一种主机，它们是一台特定的机器。另一方面，主机可以指提供托管服务来维护多个 Web 服务器的整个组织。从这个意义上说，您可以从主机运行服务器。

HTTP协议：超文本传输协议。Web 浏览器和 Web 服务器用于通过 Internet 相互通信的协议。
网址：统一资源定位器。URL 标识特定的 Web 资源。一个简单的例子是https://github.com/someone。URL 指定协议（“https”）、主机名（github.com）和文件名（某人的个人资料页面）。用户可以通过HTTP从域名为 github.com 的网络主机获取此URL标识的Web资源。（绕口令很多？！

从代码到网页的旅程
好了，现在我们已经掌握了基本定义，让我们浏览一下 Github 搜索，看看我们如何从在地址栏中键入的 URL 转到正在运行的网页：
在浏览器中键入 URL

浏览器解析 URL 中包含的信息。这包括协议 （“https”）、域名 （“github.com”） 和资源 （“/”）。在这种情况下，“.com”之后没有任何内容来指示特定资源，因此浏览器只知道检索主（索引）页面

浏览器与您的 ISP 通信，以对托管www.github.com的 Web 服务器的 IP 地址进行 DNS 查找。DNS 服务将首先联系根名称服务器，该服务器查看https://www.github.com并回复“.com”顶级域的名称服务器的 IP 地址。此地址将发送回您的 DNS 服务。DNS 服务对“.com”名称服务器进行另一次外联，并要求它提供https://www.github.com地址。

ISP 收到目标服务器的 IP 地址后，会将其发送到您的 Web 浏览器

您的浏览器从 URL 中获取 IP 地址和给定的端口号（HTTP 协议默认为端口 80，HTTPS 默认为端口 443）并打开 TCP 套接字连接。此时，您的 Web 浏览器和 Web 服务器终于连接好了。

您的 Web 浏览器向 Web 服务器发送 HTTP 请求，以获取www.github.com 的主 HTML 网页。

来自客户端的 GET 请求

Web 服务器接收请求并查找该 HTML 页面。如果该页面存在，则 Web 服务器会准备响应并将其发送回您的浏览器。如果服务器找不到请求的页面，它将发送 HTTP 404 错误消息，代表“找不到页面”。

服务器响应

您的 Web 浏览器获取它收到的 HTML 页面，然后对其进行解析，从头到脚进行全面扫描，以查找列出的其他资产，例如图像、CSS 文件、JavaScript 文件等。

index.html 页面

对于列出的每个资产，浏览器会重复上述整个过程，为每个资源向服务器发出额外的 HTTP 请求。

浏览器加载完 HTML 页面中列出的所有其他资产后，该页面最终将加载到浏览器窗口中，并且连接将关闭

跨越互联网的深渊
值得注意的一件事是，当您提出信息请求时，信息是如何传输的。当您发出请求时，该信息被分解为许多称为数据包的小块。每个数据包都标记有一个 TCP 标头，其中包括源和目标端口号，以及一个 IP 标头，其中包含源和目标 IP 地址，以为其提供标识。然后，数据包通过以太网、WiFi 或蜂窝网络传输，并允许在任何路线上传输，并根据需要进行尽可能多的跳跃以到达最终目的地。
（我们实际上并不关心包裹是如何到达那里的——重要的是它们安全无虞地到达目的地！一旦数据包到达目的地，它们就会再次重新组装并作为一个整体交付。
那么，如何让所有数据包知道如何在不迷路的情况下到达目的地呢？
答案是 TCP/IP

TCP/IP是一个由两部分组成的系统，作为互联网的基本“控制系统”。IP 代表互联网协议;它的工作是使用每个数据包上的 IP 标头（即 IP 地址）将数据包发送和路由到其他计算机。第二部分，传输控制协议 （TCP），负责将消息或文件分解为更小的数据包，使用 TCP 标头将数据包路由到目标计算机上的正确应用程序，如果数据包在途中丢失，则重新发送数据包，并在数据包到达另一端后按正确的顺序重新组合数据包。
最终画面
但是等等——工作还没有完成！现在您的浏览器已经拥有了构成网站的资源（HTML、CSS、JavaScript、图像等），它必须经过几个步骤才能将资源作为人类可读的网页呈现给您。
您的浏览器有一个负责显示内容的渲染引擎。渲染引擎以小块的形式接收资源的内容。然后有一个 HTML 解析算法，它告诉浏览器如何解析资源。
一旦解析，它就会生成 DOM 元素的树结构。DOM 代表文档对象模型，它是如何表示位于 HTML 文档中的对象的约定。每个文档的这些对象（或“节点”）都可以使用 JavaScript 等脚本语言进行操作。

DOM 树

构建 DOM 树后，将解析样式表以了解如何设置每个节点的样式。使用这些信息，浏览器向下遍历 DOM 节点并计算每个节点的 CSS 样式、位置、坐标等。
一旦浏览器有了 DOM 节点及其样式，它终于准备好相应地将页面绘制到您的屏幕上。结果是：你在互联网上看过的所有东西。
